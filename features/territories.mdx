Territories allow you to define geographic regions, assign team members, and track performance within each area.

## Overview

Territories use GeoJSON boundaries to define geographic areas. You can assign team members to territories, track coverage metrics, and identify gaps in your geographic reach.

### Territory Structure

From `territories.ts:24-51`, territories are defined as:

```typescript
interface Territory {
  id: string;
  workspaceId: string;
  name: string;
  description?: string;
  color: string;              // hex color for map rendering (e.g. "#3b82f6")
  boundary: GeoJSON;          // Polygon or MultiPolygon
  boundarySource: "custom_draw" | "zip_code" | "city" | "county" | "state";
  isExclusive: boolean;       // If true, only one member can be assigned
  createdBy: string;
  createdAt: Date;
  updatedAt: Date;
}
```

### GeoJSON Boundary Formats

Territories support two GeoJSON geometry types:

<Tabs>
  <Tab title="Polygon">
    ```json
    {
      "type": "Polygon",
      "coordinates": [
        [
          [-122.4194, 37.7749],
          [-122.4094, 37.7849],
          [-122.3994, 37.7749],
          [-122.4094, 37.7649],
          [-122.4194, 37.7749]
        ]
      ]
    }
    ```
    
    A single continuous area. The first and last coordinate must be identical to close the polygon.
  </Tab>
  
  <Tab title="MultiPolygon">
    ```json
    {
      "type": "MultiPolygon",
      "coordinates": [
        [
          [
            [-122.4194, 37.7749],
            [-122.4094, 37.7849],
            [-122.3994, 37.7749],
            [-122.4194, 37.7749]
          ]
        ],
        [
          [
            [-122.5194, 37.8749],
            [-122.5094, 37.8849],
            [-122.4994, 37.8749],
            [-122.5194, 37.8749]
          ]
        ]
      ]
    }
    ```
    
    Multiple separate areas (e.g., non-contiguous neighborhoods).
  </Tab>
</Tabs>

<Note>
Coordinates use the WGS84 coordinate system (EPSG:4326): `[longitude, latitude]`. This is the standard used by GPS and web maps.
</Note>

### Boundary Sources

Track how the boundary was created:

- `custom_draw` - Hand-drawn by user
- `zip_code` - Based on ZIP code boundary
- `city` - City boundary
- `county` - County boundary
- `state` - State boundary

## Assigning Team Members

Assign one or more team members to a territory.

### Exclusive Territories

When `isExclusive: true`, only one member can be assigned at a time:

```typescript
await api.territories.assign.mutate({
  territoryId: territory.id,
  userIds: ["user_123"] // Only one user for exclusive territories
});
```

From `territories.ts:214-294`

### Shared Territories

When `isExclusive: false`, multiple members can be assigned:

```typescript
await api.territories.assign.mutate({
  territoryId: territory.id,
  userIds: ["user_123", "user_456", "user_789"]
});
```

<Warning>
Assigning members to an exclusive territory replaces any existing assignment. For shared territories, new assignments are added without removing existing ones.
</Warning>

### Unassigning Members

```typescript
await api.territories.unassign.mutate({
  territoryId: territory.id,
```

### How It Works

Coverage uses PostGIS spatial queries:

```sql
-- From territories.ts:390-407
SELECT COUNT(*)
FROM properties
WHERE workspace_id = ?
  AND latitude IS NOT NULL
  AND longitude IS NOT NULL
  AND ST_Within(
    ST_SetSRID(ST_MakePoint(longitude, latitude), 4326),
    ST_MakeValid(ST_SetSRID(ST_GeomFromGeoJSON(?), 4326))
  )
```

<Info>
Only properties, contacts, and deals with valid latitude/longitude coordinates are included in coverage metrics.
</Info>

## Technical Details

### Spatial Queries

Relio uses PostgreSQL with PostGIS extension for spatial operations:

- **ST_GeomFromGeoJSON**: Convert GeoJSON to PostGIS geometry
- **ST_MakeValid**: Ensure geometry is topologically valid
- **ST_SetSRID**: Set spatial reference ID (4326 = WGS84)
- **ST_Within**: Test if point is within polygon
- **ST_Union**: Combine multiple territory boundaries

From `territories.ts:391, 404, 421, 439, 507, 517`

### Database Schema

```sql
-- territories table
CREATE TABLE territories (
  id TEXT PRIMARY KEY,
  workspace_id TEXT NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  description TEXT,
  color TEXT NOT NULL DEFAULT '#3b82f6',
  boundary JSONB NOT NULL,  -- GeoJSON
  boundary_source TEXT NOT NULL DEFAULT 'custom_draw',
  is_exclusive BOOLEAN NOT NULL DEFAULT true,
  created_by TEXT NOT NULL REFERENCES users(id),
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
  
  UNIQUE(workspace_id, name)
);

-- territory_assignments table
CREATE TABLE territory_assignments (
  id TEXT PRIMARY KEY,
  territory_id TEXT NOT NULL REFERENCES territories(id) ON DELETE CASCADE,
  member_id TEXT NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  assigned_by TEXT NOT NULL REFERENCES users(id),
  assigned_at TIMESTAMP NOT NULL DEFAULT NOW(),
  
  UNIQUE(territory_id, member_id)
);
```

### Assignment Rules

From `territories.ts:258-291`:

1. Verify territory exists and belongs to workspace
2. Resolve user IDs to member IDs within workspace
3. If exclusive, enforce single assignee limit
4. If exclusive, delete existing assignments first
5. Insert new assignments (ignore conflicts for shared)

## Limitations & Considerations

<Warning>
**Territory limitations:**

- Requires PostgreSQL with PostGIS extension
- Only records with valid lat/long are included in coverage
- Large territories with complex boundaries may be slow to query
- Territory names must be unique within a workspace
- Coordinate system is WGS84 (EPSG:4326) only
</Warning>
